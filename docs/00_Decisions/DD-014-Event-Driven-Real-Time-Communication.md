# Context

Smart NUB Campus provides collaborative features where users expect immediate updates without manually refreshing the application.

Examples include:

- Receiving new messages.
- Connection request notifications.
- Team invitations.
- Reputation updates.
- AI task completion.
- Discussion replies.
- Resource approvals.

Traditional request-response communication is insufficient for these interactions because updates should appear as soon as events occur.

---

# Decision

Smart NUB Campus shall adopt an **event-driven communication architecture** for all real-time features.

The initial implementation shall use **Socket.io** to deliver events between the client and the server.

The architecture should remain flexible so that another real-time transport technology can replace Socket.io in the future without changing the application's business logic.

---

# Event-Driven Architecture

Business events generated by the backend shall trigger real-time notifications to subscribed clients.

Examples include:

- User Connected
- Connection Request Sent
- Connection Accepted
- New Message
- Message Read
- Team Invitation Sent
- Team Invitation Accepted
- Discussion Reply Added
- Question Answered
- Resource Approved
- Reputation Awarded
- AI Task Completed
- System Notification Created

The application should react to these events rather than relying on continuous polling.

---

# Initial Implementation

The first implementation shall use:

- Socket.io Server
- Socket.io Client

Socket.io provides:

- Automatic reconnection
- Event-based messaging
- Room support
- Broadcast capabilities
- Cross-browser compatibility

---

# Rationale

An event-driven architecture improves responsiveness and user experience.

Benefits include:

- Immediate updates.
- Reduced unnecessary API polling.
- Better scalability for collaborative features.
- Cleaner separation between business logic and communication.
- Easier addition of future real-time features.

---

# Consequences

## Positive

- Real-time user experience.
- Lower network overhead compared to polling.
- Highly extensible architecture.
- Better responsiveness.
- Cleaner event-based programming model.

## Negative

- Increased backend complexity.
- Requires connection lifecycle management.
- Requires additional infrastructure for large-scale deployments.

---

# Alternatives Considered

## HTTP Polling

Continuously request updates from the server.

**Rejected because:**

- Generates unnecessary network traffic.
- Slower user experience.
- Less scalable.

---

## Server-Sent Events (SSE)

Use one-way server push.

**Rejected because:**

- Does not support bidirectional communication.
- Less suitable for interactive collaboration features.

---

## Native WebSockets

Use the browser WebSocket API directly.

**Rejected because:**

- Requires implementing reconnection, rooms, and event abstractions manually.
- Socket.io provides these capabilities out of the box for the initial version.
